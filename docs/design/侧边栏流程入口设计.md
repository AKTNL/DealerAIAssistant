# 侧边栏流程入口设计

**日期：** 2026-05-12

## 目标

将聊天页侧边栏从简单的建议问题列表升级为面向流程的入口面板，反映 `docs/yx/yx_sheji.html` 中描述的业务和 Agent 运行逻辑。

升级后的侧边栏必须：

- 将六个业务场景呈现为结构化流程卡片
- 每张卡片显示简短场景摘要、工具提示和步骤预览
- 允许用户在提问前展开卡片以检查流程
- 当用户点击问题按钮时立即发送问题

## 范围

本设计仅涵盖聊天页的前端侧边栏入口体验。

包含：

- 侧边栏卡片结构和交互模型
- 场景元数据的 i18n 数据结构变化
- 侧边栏问题的直接发送行为
- 新侧边栏行为的测试覆盖

排除：

- 后端分析规划变更
- 聊天消息渲染变更
- SSE 事件格式变更
- 新的独立工作流页面
- 助手实际运行时工具选择方式的变化

## 问题陈述

当前侧边栏已按业务主题分组建议提示词，但其行为仍像静态提示词列表：

- 卡片不解释场景
- 用户看不到隐含的分析步骤
- 工具参与情况未展示
- 点击提示词仅填充输入框，而非直接发起提问

这造成了 `yx_sheji.html`（描述明确的"问题 -> 工具 -> 逻辑"流程）与实际聊天入口体验（仅暴露问题文本）之间的不匹配。

## 设计决策

### 1. 侧边栏变为流程卡片面板

每个现有业务模块变为流程卡片，而非简单的可折叠文本组。

每张卡片展示：

- `title`：场景名称
- `summary`：一句话描述该场景分析的内容
- `tools`：以标签形式展示的简短工具标签
- `questions`：一个或多个直接发送的问题按钮

每张卡片可展开以显示：

- `steps`：分析路径的有序预览
- 可选的 `logicHint`：一句简短描述结果如何得出

这保留了当前轻量的"选择问题并提问"体验，同时暴露了底层的 Agent 导向流程。

### 2. 卡片展开和问题发送是分离的操作

为避免意外发送：

- 点击卡片头部展开或折叠流程预览
- 点击问题按钮立即发送该问题

卡片本身从不提交请求。

### 3. 侧边栏问题绕过仅填充输入框的行为

当前行为：

- 侧边栏问题点击 -> 填充 `promptInput`

新行为：

- 侧边栏问题点击 -> 调用与输入框提交操作相同的聊天提交路径

这保持了一条请求路径，保留现有验证行为，包括：

- 当请求已在发送中时阻止
- 当缺少模型设置时阻止
- 如需要则打开模型设置面板

### 4. 六个原型场景保持为顶层信息架构

侧边栏继续使用已与 `yx_sheji.html` 对齐的六个场景分组：

- 目标达成
- 漏斗转化
- 逾期跟进
- 新客 KPI
- 对标分析
- 线索来源趋势

视觉和数据模型升级必须深化这些分组，而非引入第二个导航模型。

### 5. 流程预览是描述性的，非权威运行时状态

侧边栏中显示的步骤是对用户的指导。它们不声称后端将按确切顺序执行那些确切的工具调用。

这一区分很重要，因为：

- 侧边栏是对预期流程的产品化解释
- 实际运行时规划可能独立演进

因此 UI 应使用"分析路径"或"典型步骤"等标签，而非"实时执行状态"。

## 数据模型

`frontend/src/i18n/messages.js` 中的 `promptModules` 结构应从：

```js
{
  title: "...",
  questions: ["...", "..."]
}
```

扩展为：

```js
{
  title: "...",
  summary: "...",
  tools: ["getCurrentDate", "searchDealers", "getTargets"],
  steps: [
    { title: "...", tool: "..." },
    { title: "...", tool: "..." }
  ],
  logicHint: "...",
  questions: ["...", "..."]
}
```

规则：

- 每个模块必须提供 `title`、`summary`、`tools`、`steps` 和 `questions`
- `tools` 是紧凑的展示列表，不是执行的权威来源
- `steps` 应保持简短，理想情况下每个场景 2-4 项
- `questions` 保持为纯字符串，因为发送路径只需要最终文本

## 组件设计

### `frontend/src/components/layout/ExampleSidebar.vue`

变更后的职责：

- 渲染场景卡片而非简单的手风琴行
- 追踪哪张卡片已展开
- 在折叠状态显示摘要和工具标签
- 在展开状态显示有序步骤和逻辑提示
- 立即发送被点击的问题

事件变更：

- 将当前仅填充的问题操作替换为提交风格的事件
- 保持 `new-chat`、`close` 和 `toggle-sidebar` 不变

推荐事件契约：

- `submit-prompt`

负载：

- 被点击的问题字符串

### `frontend/src/views/ChatView.vue`

`ChatView` 应停止将侧边栏问题视为填充输入框操作。

当前行为：

- 接收 `fill-prompt`
- 将问题赋值给 `promptInput`
- 聚焦输入框

新行为：

- 接收 `submit-prompt`
- 关闭移动端侧边栏
- 调用 `submitPrompt(question)`

`handleFillPrompt` 应替换为适合新契约的直接发送处理函数。

### `frontend/src/composables/useChat.js`

不需要结构性变更。

侧边栏应复用：

- `submitPrompt(promptOverride)`

这将验证、流式输出、错误处理和设置门控保持在一处。

## 交互流程

### 桌面端

1. 用户在侧边栏看到六个场景卡片。
2. 用户点击卡片头部以检查典型分析路径。
3. 用户点击该卡片内的一个问题按钮。
4. 问题立即被提交。
5. 正常聊天请求生命周期开始。

### 移动端

1. 用户从顶栏打开侧边栏。
2. 用户根据需要展开场景卡片。
3. 用户点击一个问题按钮。
4. 侧边栏关闭。
5. 问题立即被提交。

## 错误处理

- 如果 `isSending` 为 `true`，问题按钮必须像当前侧边栏提示词一样禁用。
- 如果模型设置不完整，侧边栏问题提交必须触发现有的模型设置中断流程。
- 如果提交失败，错误应继续通过现有请求错误机制出现在输入区域。
- 卡片展开即使在发送中也必须保持可用；仅提交操作需要被禁用。

## 视觉方向

侧边栏应保持与当前工作空间风格一致，但新卡片应读起来像结构化工作流模块，而非通用的提示词桶。

关键线索：

- 清晰的卡片边界
- 紧凑的工具标签
- 可见的步骤编号或步骤标记
- 问题按钮样式为可操作的入口，与卡片头部区分开

设计应优先考虑可读性而非密度，特别是因为每张卡片现在包含导航和解释性内容。

更新或新增测试以证明：

- 场景卡片仍然从 `promptModules` 渲染
- 摘要和工具元数据正确渲染
- 展开卡片可显示步骤和逻辑提示
- 点击问题以问题文本发射新的提交事件
- 发送中问题按钮被禁用
- `ChatView` 将侧边栏问题点击转发到 `submitPrompt(question)`
- 移动端侧边栏在侧边栏问题提交时仍关闭

目标测试文件：

- 保持中英文结构的语言一致性

- 保留现有侧边栏框架行为

### 修改 `frontend/src/views/ChatView.vue`

- 用直接提交处理替换仅填充的侧边栏处理

### 修改 `frontend/src/components/__tests__/WorkspaceChrome.spec.js`

## 验收标准

本设计在以下条件满足时完成：

- 侧边栏将六个业务流程呈现为结构化卡片
- 每张卡片显示摘要、工具和可展开的步骤预览
- 点击侧边栏问题立即发送，而不先填充输入框
- 卡片展开不会意外发送请求
- 直接发送行为仍然遵守现有的模型设置门控和发送状态守卫
