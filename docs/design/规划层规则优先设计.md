# 规划层规则优先设计

**日期：** 2026-05-11

## 目标

使分析规划层与 `docs/经销商AI分析助手_完整原型设计文档.docx` 对齐，将场景识别、范围提取、KPI 计算和回复塑造从开放式 LLM 工具规划路径中移出，放入确定性的后端规划层。

## 范围

本设计仅覆盖规划层差距：

- 通过专用规划路径路由分析请求
- 将分析请求分类到六个原型场景
- 从基于规则的范围检测和 KPI 计算构建结构化分析计划
- 仅使用聊天模型润色有依据的分析简报
- 对分析类回复强制执行降级和输出形态保障

本设计不涵盖：

- 前端视觉变化
- 记忆层变化
- 缺失的工具参数严格性
- API 数据集中缺失的聚合字段
- 替换当前的数据仓库或传输协议

## 问题陈述

当前后端流仍将分析问题视为带有工具访问权限的开放式模型调用。`ChatService` 仅通过 `looksLikeAnalyticsRequest()` 区分"类似分析"和"非类似分析"的请求。这与原型要求存在分歧，原型要求一个专用的规划层来：

1. 识别六个分析场景之一
2. 在执行前提取业务范围
3. 在生成回答前计算 KPI 事实
4. 输出包含追问的结构化四部分报告

代码库中已包含一个能执行六场景分类和规则驱动分析的 `RuleBasedAnalyticsService`，但该能力并非 `ChatService` 中的活跃规划路径。

## 设计决策

### 1. 分析请求使用规则优先管道

对于分析请求，规划从"LLM 决定做什么"转变为"后端决定分析什么，LLM 仅重写最终报告。"

目标流程：

`用户查询 -> 上下文组装 -> 意图识别 -> 规划 -> 数据处理 -> 报告生成 -> 流式输出`

映射实施：

- `ChatService`：请求路由、会话写入、SSE 事件发射、降级行为
- `RuleBasedAnalyticsService`：场景分类、范围检测、KPI 计算、有依据的分析计划构建
- `PromptFactory`：有依据的分析提示词构建和普通对话提示词构建
- `ChatModel`：仅对分析请求进行措辞润色

### 2. 仅两条请求路径

`ChatService` 保持恰好两条主路径：

- 分析路径
- 通用对话路径

分析路径：

1. 检测分析意图
2. 创建 `AnalyticsPlan`
3. 发送计划进度事件
4. 在 `<think>...</think>` 中发射计划可见思考
5. 使用从计划构建的有依据提示词调用已配置的模型
6. 验证塑造后的回复
7. 在任何分析模型失败时，返回计划降级回复

通用路径：

1. 保持当前对话流程
2. 保持直接模型提示词构建
3. 保持当前错误处理语义

### 3. 规划层返回结构化计划，而非仅字符串

引入专用模型对象 `AnalyticsPlan`，而非仅返回最终的分析回复字符串。

建议字段：

- `scenario`：六个分析场景枚举值之一
- `scopeSummary`：用于提示词和思考显示的人类可读范围摘要
- `progressMessages`：分析请求的有序 SSE 进度文本
- `visibleThinking`：确定性的三步可见思考文本
- `groundedReference`：面向模型的事实来源
- `fallbackReply`：已满足输出要求的完整用户可读 Markdown 回复

`groundedReference` 必须包含足够结构，使聊天模型能在不虚构事实的情况下重写。应包括：

- 场景名称
- 提取的范围
- 涉及的数据集
- 计算的 KPI 事实
- 表格行或行摘要
- 草稿结论
- 草稿简短分析
- 恰好两个追问候选项

### 4. 六个场景是一等规划概念

规划层必须将分析请求分类到原型定义的六个场景：

- `TARGET_ACHIEVEMENT`（目标达成）
- `OPPORTUNITY_FUNNEL`（商机漏斗）
- `SALES_FOLLOW_UP`（销售跟进）
- `CAMPAIGN_PERFORMANCE`（活动效果）
- `DEALER_BENCHMARK`（门店对标）
- `LEAD_SOURCE`（线索来源）

`RuleBasedAnalyticsService` 已包含这些场景的主题检测。该现有逻辑成为分析请求的权威规划分类器。

### 5. 分析输出形态由后端强制执行

每条分析回复，无论由模型润色生成还是直接降级生成，都必须满足原型输出结构：

- `## 结论`
- `## 数据支持`
- `## 简短分析`
- `FOLLOW_UP_QUESTIONS:`

附加规则：

- `## 数据支持` 必须包含 HTML `<table>` 输出，而非 Markdown 表格
- `FOLLOW_UP_QUESTIONS:` 必须包含恰好两个编号项
- 模型不得暴露工具名称、API 名称、"参考事实"或内部规划实现细节

如果润色后的模型回复违反这些规则，`ChatService` 必须丢弃它并返回 `fallbackReply`。

### 6. 分析路径不再依赖开放式工具规划

对于分析请求，`ChatService` 不应依赖当前开放式的 `toolCallbacks(aiToolCallbackProvider)` 路径来确定数据集或计算。

这意味着规划层（而非模型）变为负责：

- 场景选择
- 范围解析
- KPI 计算
- 报告结构

模型仅继续负责：

- 可读性
- 语气
- 措辞
- 简洁综合而不改变事实

## 详细流程

### 分析请求

1. `ChatService` 验证模型设置。
2. `ChatService` 检测请求语言。
3. `ChatService` 确定请求为分析类。
4. `ChatService` 将用户消息添加到会话记忆中。
5. `ChatService` 调用 `RuleBasedAnalyticsService.plan(message, language)` 或等效方法。
6. 规划服务返回 `AnalyticsPlan`。
7. `ChatService` 将每个 `progressMessages` 项写入 `event: progress`。
8. `ChatService` 将包裹在 `<think>...</think>` 中的 `visibleThinking` 发射出去。
9. `ChatService` 使用 `PromptFactory.buildGroundedModelPrompt(...)` 调用已配置的模型。
10. `ChatService` 验证结果回复结构。
11. 如果有效，回复被流式输出并存储。
12. 如果无效或模型调用失败，则转而流式输出并存储 `fallbackReply`。
13. `ChatService` 发射 `done`。

### 非分析请求

1. `ChatService` 验证模型设置。
2. `ChatService` 检测语言。
3. `ChatService` 确定请求不是分析类。
4. 现有对话提示词流程保持不变。

## 提示词规则

`PromptFactory.buildGroundedModelPrompt(...)` 必须明确告知模型：

- 参考事实是唯一的事实来源
- 事实、KPI 值和表格行不得更改
- 最终回答必须保留四个必填部分
- `## 数据支持` 必须使用 HTML `<table>`
- 回答必须以恰好两个追问结束，位于 `FOLLOW_UP_QUESTIONS:` 下
- 不得提及内部实现细节

如果模型以英文调用，相同结构仍然适用，但如果产品要求一致渲染，章节标题可保持原型对齐。实现应使后端验证器与所选标题策略对齐。

## 失败策略

分析请求必须优先返回确定性答案而非模糊的模型失败。

失败处理规则：

- 如果规划失败，返回清晰的分析失败响应，不降级到无约束的模型推理
- 如果规划成功但模型润色失败，返回 `fallbackReply`
- 如果模型润色返回无效结构，返回 `fallbackReply`
- 如果请求是非分析类，保持当前对话失败行为

## 文件级变更计划

### 修改 `backend/src/main/java/com/brand/agentpoc/service/ChatService.java`

- 拆分分析和通用对话路径
- 对分析请求调用规划服务
- 发射计划进度和计划可见思考
- 对分析请求使用有依据的提示词构建
- 验证润色后的分析回复
- 应用降级回复处理

### 修改 `backend/src/main/java/com/brand/agentpoc/service/RuleBasedAnalyticsService.java`

- 将服务从"返回最终字符串回复"演进为"返回结构化分析计划"
- 保留现有六场景分类和 KPI 逻辑
- 构建 `groundedReference` 和 `fallbackReply`

### 修改 `backend/src/main/java/com/brand/agentpoc/ai/PromptFactory.java`

- 强化有依据的分析提示词规则
- 保留通用对话提示词路径

### 新增 `backend/src/main/java/com/brand/agentpoc/service/AnalyticsPlan.java`

- 定义规划层返回的结构化分析计划

### 修改 `backend/src/test/java/com/brand/agentpoc/service/ChatServiceTest.java`

- 覆盖分析路径路由
- 覆盖模型润色降级
- 覆盖无效润色回复的拒绝
- 验证非分析流程保持直接

### 新增或修改 `backend/src/test/java/com/brand/agentpoc/service/RuleBasedAnalyticsServiceTest.java`

- 覆盖六场景分类
- 覆盖结构化计划输出
- 验证降级回复包含必填部分

## 测试策略

此变更的测试应证明：

- 分析请求不再依赖开放式规划
- 分析回复在模型润色失败时也能成功
- 润色回复在破坏结构时被拒绝
- 非分析请求仍使用现有对话流程
- 每个场景仍映射到确定性的规划输出

最低测试用例：

- 分析请求路由到规划服务并使用有依据的提示词
- 分析请求在模型抛出异常时返回降级回复
- 分析请求在模型省略 `FOLLOW_UP_QUESTIONS:` 时返回降级回复
- 分析请求在 `## 数据支持` 缺少 HTML `<table>` 时返回降级回复
- 非分析请求绕过规划服务
- 代表性提示词映射到六个场景中的每一个

## 超出范围的后续工作

在本次规划层工作完成后，以下差距可单独处理：

- 将 `getCurrentDate` 添加为真实工具或显式规划输入
- 收紧工具参数要求以匹配原型
- 添加缺失的聚合字段，如 `totalCount`、`campaignCount` 和 `totalTaskCount`
- 修订前端措辞或样式差异

## 验收标准

本设计在以下条件满足时完成：

- 分析请求通过确定性的六场景后端路径进行规划
- 分析回复在模型润色前基于规则计算的事实
- 分析回复始终满足所需的四部分结构，通过有效模型润色或后端降级
- 非分析对话行为保持完整
