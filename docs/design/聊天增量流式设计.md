# 聊天增量流式设计

**日期：** 2026-05-13

## 目标

将聊天回复渲染从"在结束时返回一个完整的助手回复"改为"随着数据到达逐步返回助手回复片段"。

本设计针对当前差距：应用已使用 SSE 传输，但可见回复正文仍然在模型调用完成后一次性全部出现。

## 范围

本设计仅涵盖聊天回复的增量正文流式输出。

包含：

- 助手正文内容的后端流式行为
- 流式正文片段的前端增量累积和渲染
- 增量流式更新的滚动锁定行为
- 保留当前 `progress`、`thinking`、`message`、`done` 和 `error` 事件类型
- 流式回复的会话记忆持久化时机
- 异常长的流式回复的后端安全保护
- 后端和前端流式行为的回归覆盖

排除：

- 更改同步 `/api/chat` 行为
- 添加片段缓冲或句子级聚合
- 重新设计 Markdown 渲染
- 更改流式交付之外的分析规划或降级逻辑

## 当前问题

当前技术栈仅部分流式化：

- 后端将 `/api/chat/stream` 暴露为 SSE
- 前端增量消费响应正文
- 但 `ChatService.streamChat(...)` 仍然阻塞直到产生完整回复字符串
- 后端随后发射包含整个助手正文的单个最终 `message` 事件
- 前端在每次 `message` 时替换助手内容，这假设是单个完整负载，而非多个增量片段

结果，用户首先看到进度指示器，然后完整的助手回答一次性出现，而不是逐步构建。

## 设计决策

### 1. 原样使用上游片段

后端应按模型输出的自然流式片段边界转发，而非缓冲和重组内容。

这是推荐的第一步，因为：

- 最小化实现复杂性
- 减少端到端延迟
- 保留从上游模型到 UI 的最自然流式路径

权衡是片段边界可能不与句子或 Markdown 块对齐。这对于本阶段是可接受的。

### 2. 保持现有 SSE 事件契约

当前事件名称保持不变：

- `progress`
- `message`
- `done`
- `error`

唯一的语义变化是 `message` 变为增量助手正文片段，而非完整的最终正文。

这保留了大部分当前的前端/后端契约，避免了仅为分块正文交付引入并行事件类型。

### 3. 保持 `thinking` 为独立的单次消息

可见思考块应保持其当前行为：

- 在正文流开始前发送一次
- 与流式助手正文保持分离
- 当前可折叠 UI 继续工作，无需增量推理组装

这使变更聚焦在用户可见的主要问题上。

即使此阶段保持 `thinking` 为单次消息，后端实现应保留扩展路径，以便后续 `thinking` 可以作为多个片段发射，而无需重新设计事件写入器或前端事件路由器。

### 4. 前端追加而非替换

前端应将每个非思考 `message` 事件视为正文增量：

- 将片段追加到 `assistantMessage.content`
- 从更新后的累积内容重新渲染 `assistantMessage.html`
- 在片段持续到达时保持消息可见

这是将现有 SSE 解析器转变为真正增量聊天显示所需的最小变更。

### 5. 仅在用户已在底部时自动滚动

如果用户已向上滚动以阅读早期内容，增量片段不得强制视口回到底部。

前端应追踪用户当前是否锚定在聊天视口底部：

- 如果已锚定，新片段继续自动滚动
- 如果未锚定，自动滚动暂停
- 暂停时，UI 应暴露轻量"新消息"图标，让用户可以跳回底部

这防止流式输出牺牲手动阅读行为。

### 6. 为频繁片段更新添加轻量性能降级

第一个实现可以在每个片段上重新渲染。

如果片段频率导致可见卡顿，前端应有直接的优化路径：

- 通过 `requestAnimationFrame` 批处理视觉更新
- 或将 DOM 更新限制为最新助手消息框架，而非触发更广泛的响应式工作

这不需要前期构建为复杂的优化层，但实现应避免使未来调整变得困难。

### 7. 仅在流完成后持久化助手历史

后端应在服务端累积流式正文片段，并仅在成功完成后将最终助手回复存储在会话记忆中。

这避免了：

- 流被中断时保存部分回复
- 同步和流式聊天路径之间的历史分歧
- 以不完整内容污染会话记忆

如果流以 `error` 结束，助手回复不应被持久化。

### 8. 强制执行硬性服务端回复长度上限

后端应对流式助手输出强制执行硬上限，以便意外长的模型回复无法无限制地增长内存累积缓冲区。

此安全保护应：

- 在流式传输期间对累积的助手正文总长度设置上限
- 一旦达到上限就停止进一步累积
- 以明确的失败路径或截断策略干净地终止流

确切的上限值可在实现期间设置，但设计要求硬限制存在。

## 后端设计

### `ChatService.streamChat(...)`

该方法应停止为正常流式模型回复调用阻塞的完整回复路径。

预期行为：

1. 同今天一样检测语言和分析模式。
2. 在模型调用前发射第一个 `progress` 事件。
3. 对于配置缺失和基于规则降级的情况，继续使用现有的单次回复路径。
4. 对于已配置的模型调用，切换到流式模型调用。
5. 发射当前可见思考有效负载一次。
6. 将每个上游正文片段作为单独的 `message` 事件转发。
7. 将这些正文片段累积到本地缓冲区。
8. 如果超过配置的回复长度上限则停止累积。
9. 成功完成后，发射 `done` 并持久化累积的完整助手回复。
10. 失败时，发射 `error` 并跳过助手持久化。

### 流式模型调用

已配置模型路径应使用 Spring AI 的流式聊天 API，而非当前的阻塞 `.call().content()` 路径。

实现应：

- 完全同今天一样构建提示词
- 保留分析依据和验证行为
- 仅流式输出助手正文文本
- 避免更改同步 `chat(...)` 代码路径
- 保持流式写入路径足够通用，使 `thinking` 片段后续可通过同一事件管道增量发射

对于分析请求，流式输出仍需经过现有提示词契约。本设计不在流期间添加增量结构验证；验证仍仅为非流式路径的完成阶段关注点。

### 错误处理

如果在已发射一些片段后流式输出失败：

- 发射 `error`
- 不发射 `done`
- 不在会话记忆中持久化助手回复

前端将保留已显示的任何正文内容。

如果因回复长度上限超限而停止流式输出，应遵循同一非成功路径，而非假装流正常完成。

## 前端设计

### `useChat`

`onEvent` 处理逻辑应区分：

- 包裹在 `<think>...</think>` 中的思考有效负载
- 增量助手正文片段

对于正文片段：

- 追加到 `assistantMessage.content`
- 重新运行 Markdown 渲染
- 保持 `assistantMessage.streaming = true` 直到 `done` 或 `error`

这意味着当前 `applyAssistantPayload(...)` 行为需要一个流式正文追加路径，而非假设每次都是完整的标准化有效负载。

聊天组合函数还应停止在每个片段上无条件滚动到底部。而应：

- 检测视口是否已接近底部
- 仅在该锁活跃时自动滚动
- 当流式输出继续离开屏幕时，为 UI 暴露小型未读/新内容状态

### 部分片段期间的 Markdown 渲染

流式输出期间部分 Markdown 是可接受的。

渲染器可能临时接收到：

- 不完整的强调标记
- 半完成链接
- 未完成的表格或列表

这是可接受的，因为每个后续片段将重新渲染整个累积正文并自然收敛到正确的最终 HTML。

此阶段不需要额外的客户端缓冲。

### 错误行为

如果在一些正文片段后到达 `error` 事件：

- 保留累积的助手内容可见
- 停止加载状态
- 同今天一样显示用户友好的错误消息

UI 不应清除或替换部分回复。

### 性能行为

默认行为可在每个片段上立即渲染。

如果这在真实片段节奏下导致可见卡顿，预期的降级方案是：

- 将到达的片段累积到等待缓冲区
- 在 `requestAnimationFrame` 期间刷新可见追加

这提供了一个简单的节流点而不改变事件契约。

## 测试策略

### 后端

添加验证以下内容的测试：

- 流式已配置模型回复在上游流产生多个片段时发射多个 `message` 事件
- 助手会话记忆仅在成功流完成后写入
- 失败的流式已配置模型回复发射 `error` 且不持久化助手历史

现有的测试：

- 缺失模型设置指导
- 分析降级行为
- 初始进度发射

应保持有效，只需最小调整。

### 前端

添加验证以下内容的测试：

- 多个 `message` 事件追加到一个助手回复中，而非替换先前内容
- 在增量正文更新期间 `streaming` 保持为 `true`
- 部分正文片段后的 `error` 保留可见累积内容并结束加载状态
- 自动滚动仅在视口处于底部锁定时继续
- 用户手动向上滚动后新片段不强制滚动视口
- 当自动滚动暂停时新内容指示器状态出现

## 验收标准

本设计在以下条件满足时完成：

- 已配置的流式聊天回复在 UI 中逐步出现，而非一次性全部出现
- 每个流式正文片段在前端按顺序追加
- 聊天视图仅在用户已在底部时自动滚动
- 向上手动滚动不被新流式片段中断
- 当前 `progress`、`thinking`、`done` 和 `error` 行为仍正常工作
- 后端有防止无界流式回复累积的硬上限
- 会话记忆仅在成功流完成后存储最终助手回复
- 中断的流保持已显示内容可见但不持久化助手历史
- 回归测试覆盖后端和前端的新流式片段行为
